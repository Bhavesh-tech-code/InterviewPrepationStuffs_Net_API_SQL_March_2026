Basics & Fundamentals (Q1–Q15)
Q1. Web API क्या है?
Answer:
Web API (Application Programming Interface) एक HTTP-based service है जो data को JSON, XML या अन्य formats में expose करती है।
 इसका main purpose है कि अलग-अलग platforms (web, mobile, desktop) एक ही data का इस्तेमाल कर सकें।

📌 Use:
Mobile app और Web app एक ही data source से data ले सकते हैं।
Microservices में services के बीच communication।
📌 Example:
अगर आपके पास एक “Products” database है, तो Web API एक URL /api/products देगी, जिस पर GET request करने पर आपको product list JSON में मिलेगी।

Q2. ASP.NET Core Web API और ASP.NET Web API में अंतर?
Answer:
ASP.NET Web API → पुराना framework, केवल Windows पर चलता है, heavyweight है।
ASP.NET Core Web API → Cross-platform (Windows, Linux, Mac), lightweight, fast, modular architecture, और built-in dependency injection support देता है।
📌 Why Core is better?
क्योंकि ये modern architecture, high performance, और maintainable code देता है।

Q3. REST API क्या है?
Answer:
REST का मतलब है Representational State Transfer।
यह एक architectural style है जिसका use web services / APIs बनाने के लिए होता है।
REST API basically ek ऐसा interface है jiske through ek client (browser, mobile app, frontend),
aur ek server (backend, database) आपस में HTTP methods ke through बात करते हैं।
--REST API कैसे काम करता है?
REST API ka base communication HTTP protocol होता है।
हर resource (जैसे Student, Product, Employee) ek URI/endpoint ke रूप में represent होता है।
Example: /api/students/1 → student with ID = 1

Client server से बात करता है HTTP methods ka use करके:
-----------------------------
GET /api/students → सारे students की list लाओ।  ----data पढ़ना (read)
GET /api/students/5 → ID = 5 वाले student की detail लाओ।    ----------data पढ़ना (read) on the basis of Id.
POST /api/students → नया student create करो।   -----------------------नया data बनाना (create)
PUT /api/students/5 → ID = 5 वाले student को पूरी तरह update करो।    ----पुराना data update करना (full update)
PATCH → पुराना data update करना (partial update)
DELETE /api/students/5 → ID = 5 वाले student को delete करो।  ----------data हटाना

--------------------------------
RESTful API guidelines mainly revolve around —
✔️ Client-Server separation
✔️ Stateless communication
✔️ Proper use of HTTP methods & URIs
✔️ Resource-based design
✔️ Uniform interface
✔️ JSON/XML representation
✔️ Caching & Layered system
---------------------------------------------
1. Stateless →  				--server does not store the state of the client, all requests are independent to each other,
								--Each request must contails all information. 
								--Server request ke बीच में client की state (session info) नहीं रखता।
								--अगर client को authentication चाहिए, तो हर request में token/credentials bhejna पड़ेगा।
								
2. Client-Server Architecture → --Seperation of concerns between client and server,
                                --means client handles UI and server handles data only.	
								--Client सिर्फ request करेगा और server सिर्फ response देगा।
								
3. Uniform Interface →          --Use standard http verbs and status codes means same rules will be applied on every endpoints.
								--API का interface (endpoints, resources, methods) एक standard तरीके से होना चाहिए:
                                --Resources ko nouns (plural) से represent करो → /api/students, /api/products
                                --HTTP methods ka सही use करो:
                                --GET → read
								--POST → create
								--PUT → full update
								--PATCH → partial update
								--DELETE → delete
								--Responses standard format (JSON/XML) में दो।

4. Cacheable					--Response ko cacheable बनाओ जहाँ possible हो, ताकि बार-बार same data fetch na करना पड़े।
                                --Example: Student list ko Cache-Control header ke साथ bhejna.			
                                  
5. Layered System               --API architecture layers me हो सकती है (gateway, load balancer, server, database)।
                                --Client ko ये नहीं पता होना चाहिए कि वो direct server se baat kar raha hai ya ek proxy/load balancer ke through।

Middleware — Short, Complete Answer

"Middleware is a software component in ASP.NET Core that executes between every HTTP request and response pipeline.
 It processes HTTP requests and/or responses.
 Middleware can perform tasks like
 authentication, logging, exception handling, routing, etc.
 Each middleware can either process the request and pass it to the next middleware,
 or short-circuit the pipeline and return a response directly.


ASP.NET Core Middleware Execution Order Table
Order	Middleware Name   										    Purpose
1		app.UseDeveloperExceptionPage()/UseExceptionHandler()     	Error handling (detailed in dev, generic in production).
2		app.UseHsts()	                                                Adds HTTP Strict Transport Security headers.
3		app.UseHttpsRedirection()	                                Redirect HTTP requests to HTTPS.
4		app.UseForwardedHeaders()	                                Handles proxy headers (when behind reverse proxy).
5		app.UseRequestLocalization()	                                Sets request culture and localization settings.
6		app.UseStaticFiles()                                      	Serves static files (CSS, JS, images, etc.).
7		app.UseCookiePolicy()                                     	Configures cookie behavior and compliance.
8		app.UseRouting()	                                        Matches request to route endpoints.
9		app.UseCors()	                                                Enables Cross-Origin Resource Sharing policies.
10		app.UseAuthentication()                                    	Validates authentication credentials (JWT, cookies, etc.).
11		app.UseAuthorization()	                                        Checks if the authenticated user is authorized.
12		app.UseSession()                                           	Enables session state.
13		app.UseResponseCaching()                                        Caches responses for performance improvement.
14		app.UseWebSockets()	                                        Enables WebSocket communication.
15		app.UseStatusCodePages()	                                Returns custom responses for HTTP error status codes.
16		app.UseEndpoints()	                                        Executes endpoint matched by routing.

Important Notes::::::::::::::::
Ye recommended order hai. 
Tum apne project requirements ke hisaab se order change kar sakte ho, 
lekin kuch rules follow karne zaroori hote hain:
Exception handling → sabse pehle
HSTS/HTTPS redirection → early
Static files → routing se pehle (agar static files ko bypass karna ho routing se)
Routing → authentication/authorization se pehle
Authentication → authorization se pehle
Endpoints → last

Custom Middleware Example-- (logging, request modification, error handling)
This middleware will log the request time(kitna time laga request process hone me).

1. Create Custom Middleware Class
using Microsoft.AspNetCore.Http;
using System.Diagnostics;
using System.Threading.Tasks;

public class RequestTimingMiddleware
{
    private readonly RequestDelegate _next;

    public RequestTimingMiddleware(RequestDelegate next)
    {
        _next = next;
    }

    public async Task InvokeAsync(HttpContext context)
    {
        var stopwatch = Stopwatch.StartNew();

        // Before passing to next middleware
        await _next(context);

        stopwatch.Stop();

        var elapsedTime = stopwatch.ElapsedMilliseconds;
        var path = context.Request.Path;

        // Log the request time (for demo: console)
        Console.WriteLine($"Request to {path} took {elapsedTime} ms");
    }
}

-----------------------
2. Register Middleware in Pipeline

var builder = WebApplication.CreateBuilder(args);
var app = builder.Build();

app.UseMiddleware<RequestTimingMiddleware>();

app.MapGet("/", () => "Hello World!");

app.Run();

--------------------------------------------
3. How It Works
InvokeAsync method request handle karta hai.
_next(context) call se request next middleware ya controller ko pass hoti hai.
Response ke baad time calculate karke log karte hain.

Example Output (Console)
Request to / took 23 ms
Request to /api/products took 45 ms


✅ Interview Tip:
Agar interviewer poochta hai, tum bol sakte ho:

"This middleware measures request execution time by using Stopwatch. 
It runs before and after other middleware/controllers by calling _next(context) in between."

What are Filters in Web API?

Filters छोटे reusable components होते हैं जो ASP.NET Web API pipeline में request या response के processing के दौरान execute होते हैं।
Filters का मुख्य purpose है cross-cutting concerns handle करना जैसे:
Logging
Authentication / Authorization
Error handling
Caching
Validation

Agar mujhse koi interview mai puchta hai ki web api mai filters ke type btao to kon konse batane hai short mai btao'
SP.NET Core Web API में Filters के Main Types:

Authorization Filters → request authorize करने से पहले चलाते हैं।
Resource Filters → action method execution से पहले और बाद में, caching या resource loading के लिए।
Action Filters → action method के पहले और बाद में custom logic execute करते हैं।
Exception Filters → exceptions handle करने के लिए।
Result Filters → action के बाद, response भेजने से पहले run होते हैं।

🔹 Short Answer for Interview (ASP.NET Core):
Authorization, Resource, Action, Exception, Result filters।

Q32. Unit of Work Pattern क्या है?
Answer:
Multiple repository operations को एक ही transaction में handle करना ताकि सभी success/fail एक साथ हों।

Unit of Work Pattern एक design pattern है जो mainly database transactions को manage करने के लिए use होता है, खासकर जब हम Entity Framework या Repository Pattern use करते हैं।

सरल भाषा में:

👉 Unit of Work का काम है multiple database operations को एक single transaction की तरह manage करना।

मतलब अगर हम Insert, Update, Delete जैसे कई operations कर रहे हैं तो सब एक साथ commit होंगे, और अगर कहीं error आया तो सब rollback हो जाएंगे।

1️⃣ Real Life Example

मान लो बैंक में:

Account A से ₹1000 deduct हुआ
Account B में ₹1000 add हुआ

अगर A से deduct हो गया लेकिन B में add नहीं हुआ तो problem हो जाएगी।

इसलिए दोनों operations एक transaction में होने चाहिए।

यही काम Unit of Work Pattern करता है।

2️⃣ Unit of Work Pattern कैसे काम करता है

Steps generally ऐसे होते हैं:

Multiple repositories create होते हैं
सभी operations perform होते हैं
आखिर में Save() या Commit() call होता है

तब जाकर database में changes save होते हैं।

3️⃣ Basic Structure
IUnitOfWork Interface
public interface IUnitOfWork
{
    IStudentRepository Students { get; }
    ICourseRepository Courses { get; }

    void Save();
}
UnitOfWork Class
public class UnitOfWork : IUnitOfWork
{
    private readonly StudentContext _context;

    public IStudentRepository Students { get; private set; }
    public ICourseRepository Courses { get; private set; }

    public UnitOfWork(StudentContext context)
    {
        _context = context;
        Students = new StudentRepository(_context);
        Courses = new CourseRepository(_context);
    }

    public void Save()
    {
        _context.SaveChanges();
    }
}
4️⃣ Controller Example
public class StudentController : Controller
{
    private readonly IUnitOfWork _unitOfWork;

    public StudentController(IUnitOfWork unitOfWork)
    {
        _unitOfWork = unitOfWork;
    }

    public IActionResult Create(Student student)
    {
        _unitOfWork.Students.Add(student);
        _unitOfWork.Save();

        return Ok();
    }
}

Benefits:

1️⃣ Transaction Management
2️⃣ Multiple repositories को coordinate करता है
3️⃣ Code clean और maintainable बनता है
4️⃣ Database calls कम होते हैं

1️⃣ ACID Properties क्या हैं

ACID database transaction की fundamental properties हैं।

ACID =

A – Atomicity → या तो पूरा transaction होगा या कुछ भी नहीं
C – Consistency → database हमेशा valid state में रहेगा
I – Isolation → एक transaction दूसरे को disturb नहीं करेगा
D – Durability → commit होने के बाद data permanent रहेगा

Example:

अगर बैंक transaction में

A से ₹1000 deduct
B में ₹1000 add

और बीच में error आ गया

तो Atomicity ensure करेगी कि दोनों operations rollback हो जाएं।

Unit of Work Pattern एक application level design pattern है।

इसका काम है:

👉 multiple database operations को एक transaction के अंदर execute कराना।

Example:

Insert Student
Update Course
Delete Record

ये सब operations एक ही SaveChanges() में commit होंगे।

ToList() → result list में convert

Q37. Authorization क्या है? and what is Role-based Authorization in web api

----Verified user को resource access करने की permission देना।

----A user must first authenticate (prove who they are, usually via username/password, token, etc.).
After authentication, the system checks the roles assigned to that user (like Admin, Manager, User, Student, etc.).
Based on the role, the user is authorized (or denied) to access certain APIs or perform certain actions.

How it Works
Authentication:
User logs in → System validates credentials → Issues a token (JWT or cookie).
The token contains the user’s claims, including their roles.
Example:
{
  "sub": "12345",
  "name": "Bhavesh",
  "role": "Admin"
}

Authorization:
When user calls an API, Web API checks if the user’s role matches the required role.

Example in ASP.NET Core Web API
[Authorize(Roles = "Admin")]
[HttpGet("GetAllUsers")]
public IActionResult GetAllUsers()
{
    return Ok("Only Admins can see this!");
}

If the logged-in user has role Admin, they can access this API.
If the role is User, they will get 403 Forbidden.

🔹 Multiple Roles Example
[Authorize(Roles = "Admin,Manager")]
[HttpGet("Reports")]
public IActionResult GetReports()
{
    return Ok("Admins and Managers can view reports");
}

✅ In short:
Authentication = “Who are you?”
Authorization (Role-based) = “What are you allowed to do based on your role?”

Q43. API Versioning क्यों करते हैं?
पुराने clients को support करते हुए नए changes introduce करने के लिए।


API versioning is a strategy to manage changes to your API without breaking existing clients.
When you change your API — e.g., adding new endpoints, changing response format, 
or modifying behavior — older clients should still work. Versioning solves this.

Why API Versioning is important
Backward compatibility
Better developer experience
Easier evolution of the API
Avoids breaking existing consumers

Step 1 — Create a new Web API project
Run:

cd ApiVersionDemo

dotnet add package Microsoft.AspNetCore.Mvc.Versioning

Step 3 — Configure API Versioning in Program.cs

var builder = WebApplication.CreateBuilder(args);

// Add services to the container.
builder.Services.AddControllers();

// Add API versioning
builder.Services.AddApiVersioning(options =>
{
    options.AssumeDefaultVersionWhenUnspecified = true; // if version not specified
    options.DefaultApiVersion = new Microsoft.AspNetCore.Mvc.ApiVersion(1, 0);
    options.ReportApiVersions = true; // tells client which versions are available
});

var app = builder.Build();

// Configure the HTTP request pipeline.
app.UseHttpsRedirection();
app.UseAuthorization();
app.MapControllers();

app.Run();

This is API versioning in action.
-------------------------------
Versioning basically means managing changes over time in your software so that different versions of your application,
API, or document can coexist without breaking things for users.

1. Why Versioning is Needed
Imagine you have an API used by multiple clients (mobile apps, websites, other services).
If you change something — e.g., rename a field, change output format — existing clients can break if they aren’t updated.
Versioning helps you introduce changes without breaking existing clients.

Example:
v1: GET /api/products → returns { id, name, price }
v2: GET /api/products → returns { id, name, price, category }
If versioning is not done, older clients expecting v1 format will break.

2. Types of Versioning in APIs
Here are common approaches:
Type	               			 Example URL	                   		 Pros	                Cons
URL Path Versioning	             /api/v1/products	               		 Clear & simple	         URL changes on version change
Query String Versioning          /api/products?version=1	      		 Easy to implement	     Not as clean as URL path
Header Versioning                Accept: application/vnd.myapi.v1+json	 Clean URLs	             More complex implementation
Media Type Versioning	         Based on Content-Type header	         Flexible              	 Complexity increases
Media Type Versioning

3. Versioning in Software
Versioning isn’t just for APIs. It applies to:
Libraries/Packages (NuGet packages) → semantic versioning like 1.0.0, 1.1.0, 2.0.0
Applications → version numbers track updates and changes.
Databases → migration versioning for schema changes.

Semantic Versioning (SemVer)
Most software uses this:
MAJOR.MINOR.PATCH
MAJOR → Breaking changes
MINOR → New features, backward compatible
PATCH → Bug fixes, backward compatible

Example:
1.0.0 → initial release
1.1.0 → added new feature
2.0.0 → breaking change

✅ Key takeaway:
Versioning lets you manage change without breaking existing consumers 
— especially important for APIs in production.

Q. Difference between put and post in WEB API in dot net core 
Here is the to-the-point difference between PUT and POST in .NET Core Web API, with small code examples:

PUT vs POST
Feature	POST	PUT
Purpose	Used to create a resource (insert data).	Used to update an existing resource (full update).
Idempotent (Repeat call gives same result?)	Not idempotent → calling multiple times creates multiple records.	Idempotent → calling multiple times results in same update, not duplicate.
When used	When server should generate unique ID or new record.	When client already knows ID and wants to update record.
URL structure	api/employees	api/employees/{id}
Body contains	New data to create	Full updated object of existing record
----------------------------------------------------------------------------------------------------------------------------------------------------------------------
Example in Web API
POST (Create Employee)
[HttpPost]
public IActionResult CreateEmployee(Employee emp)
{
    _context.Employees.Add(emp);
    _context.SaveChanges();
    return CreatedAtAction(nameof(GetEmployeeById), new { id = emp.Id }, emp);
}

PUT (Update Employee)
[HttpPut("{id}")]
public IActionResult UpdateEmployee(int id, Employee emp)
{
    if (id != emp.Id)
        return BadRequest("Id mismatch");

    _context.Employees.Update(emp);
    _context.SaveChanges();
    return NoContent();
}

One Line Summary

POST = Create new object

PUT = Update existing object (replace entire object)
