Access token, refresh token, jwt token same chije hai kya?
Nahi, Access Token, Refresh Token aur JWT Token same cheez nahi hote. Ye related zaroor hote hain, 
par role alag-alag hota hai. Main simple language me samjhata hoon.

1️⃣ Access Token

Access Token ka use API ko access karne ke liye hota hai.

Jab user login karta hai to server Access Token deta hai.
Is token ko request ke header me bhej kar user API access karta hai.
Ye short time ke liye valid hota hai (jaise 15 min, 1 hour).

Example:
User login → Server Access Token deta hai → User us token se API call karta hai.

2️⃣ Refresh Token

Refresh Token ka kaam naya Access Token banana hota hai.

Jab Access Token expire ho jata hai tab Refresh Token use hota hai.
Isse user ko dobara login karne ki zarurat nahi padti.
Ye long time ke liye valid hota hai (jaise 7 din, 30 din).

Flow:

User login
Server deta hai
Access Token
Refresh Token
Access Token expire ho gaya
Refresh Token se new Access Token generate
3️⃣ JWT Token

JWT ka full form:
👉 JSON Web Token

Ye token ka format/technology hai.

Matlab:

Access Token JWT format me ho sakta hai
Refresh Token bhi JWT ho sakta hai

JWT ke andar information hoti hai:

user id
role
expiry time

Example structure:

Header.Payload.Signature
📌 Simple Example

Samjho tum kisi office building me ja rahe ho:

Access Token → Gate pass (andar jane ke liye)
Refresh Token → Reception slip jisse naya gate pass mil sakta hai
JWT → Gate pass ka format/design

Token	Purpose	Expiry
Access Token	API access	Short time

2️⃣ JWT Token Generate karna

Agar login successful ho jaye to server JWT Token generate karta hai.

Example:

private string GenerateToken(string username)
{
    var claims = new[]
    {
        new Claim(ClaimTypes.Name, username)
    };

    var key = new SymmetricSecurityKey(
        Encoding.UTF8.GetBytes("MySecretKey12345"));

    var creds = new SigningCredentials(
        key, SecurityAlgorithms.HmacSha256);

    var token = new JwtSecurityToken(
        issuer: "myapi",
        audience: "myapi",
        claims: claims,
        expires: DateTime.Now.AddMinutes(30),
        signingCredentials: creds);

    return new JwtSecurityTokenHandler().WriteToken(token);
}

Yaha:

Claims → user ki information
Key → secret key
Expiry → token kitne time tak valid hoga

Header.Payload.Signature

Fir server:

header decode karta hai
payload decode karta hai

Payload me user info hoti hai:

{
 "userId": 101,
 "role": "Admin",
 "exp": 1710000000
}
3️⃣ Signature Verify hota hai (Sabse important step)

Server ke paas secret key hoti hai.

Example:

MySecretKey12345

Server kya karta hai:

Header + Payload ko read karta hai
Same algorithm use karta hai (HMACSHA256)
Secret key se naya signature generate karta hai

Fir compare karta hai:

Generated Signature == Token Signature

Agar match ho gaya → token valid

Agar match nahi hua → token invalid

Payload me exp (expiry time) hota hai.

Example:

exp = 1710000000

Server check karta hai:

Current time < expiry time

Agar expiry cross ho gayi → token invalid.

Server ye bhi verify karta hai:

Issuer → token kis server ne banaya
Audience → token kis application ke liye hai

ValidateIssuer = true
ValidateAudience = true

ValidateIssuerSigningKey = true

user claims read karta hai

Bahut achha bhai 👍
Ye interview ka classic question hai:
“Agar koi JWT token ko change (tamper) kar de to server kaise pata lagata hai?”

Iska answer Signature verification me chhupa hua hai. Step-by-step samajho.

1️⃣ JWT Token ke 3 Parts

JWT token 3 parts ka hota hai:

Header.Payload.Signature

Example structure:

eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9
.
eyJ1c2VySWQiOjEwMSwicm9sZSI6IlVzZXIifQ
.
abc123signature
Header → algorithm info
Payload → user data (claims)
Signature → security verification
2️⃣ Signature kaise banti hai

Server jab token generate karta hai tab signature banata hai.

Conceptually process:

Signature = HMACSHA256(
  base64UrlEncode(header) + "." + base64UrlEncode(payload),
  SecretKey
)

Important cheez:

👉 SecretKey sirf server ke paas hoti hai
Client ko nahi pata hoti.

3️⃣ Agar koi Payload change kar de

Suppose original payload:

{
 "userId":101,
 "role":"User"
}

Attacker isko change kar deta hai:

{
 "userId":101,
 "role":"Admin"
}

Token ka payload part change ho gaya.

4️⃣ Server Verification Process

Jab request server ke paas aati hai to server:

Token receive karta hai
Header + Payload read karta hai
Apni SecretKey se new signature generate karta hai

Phir compare karta hai:

GeneratedSignature == TokenSignature ?
5️⃣ Result
Situation	Result
Payload change nahi hua	Signature match
Payload change hua	Signature mismatch

Agar mismatch hua:

401 Unauthorized

Matlab token invalid.

6️⃣ Isliye JWT secure hai

Kyuki attacker:

payload change kar sakta hai
lekin correct signature generate nahi kar sakta

Kyuki uske paas secret key nahi hoti.

📊 Simple Flow
Client request with JWT
        ↓
Server token read karta hai
        ↓
Header + Payload se new signature generate
        ↓
Compare with token signature
        ↓
Match → valid
Mismatch → invalid
