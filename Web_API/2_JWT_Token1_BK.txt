Q1. JWT Token Kya Hai?
JWT ka full form hai JSON Web Token.
Ye ek compact, secure aur digitally signed token hota hai,
jo data ko JSON format me store karta hai,
aur client aur server ke beech authentication/authorization ke liye use hota hai.
Stateless communication → Server ko session store nahi karna padta.

Simple shabdo me:

JWT ek ID card ki tarah hai jo server aapko login ke baad deta hai.

Is card me aapki identity + kuch extra info hoti hai.

Server signature se verify karta hai ki token me koi chhed-chhad na hui ho.

1️. Header-->Token ka metadata store hota hai (algorithm + type).

2️. Payload
Actual data (claims) hota hai, jaise userId, role, etc.

Header + Payload ko secret key se sign kiya jata hai.
Ye ensure karta hai ki token me koi modification na hua ho.

JWT Workflow
1) First Request (Login Request)
Client/Browser sends a request to the Server with User Credentials
(username + password).
Server validates these credentials from database.
If valid, the server generates a JWT (JSON Web Token) which is:
Compact
Digitally Signed
Secure
Server returns the JWT to the client.
Client stores this token in Local Storage or Session Storage (or sometimes Cookies).

2) Next / Subsequent Requests (Authorized Requests)
Client sends another request to the server and this time sends the JWT token inside HTTP header:
Authorization: Bearer <JWT-Token>
Server verifies the token using the same Secret Key used while signing the token.
If token is valid:
Server processes the request
Returns response/data with appropriate status code (200 OK, etc.)
If token is invalid / expired / tampered:
Server returns 401 Unauthorized.

One Line Summary
JWT is used to authenticate the user once, and then every request afterwards is authorized using the token, 
instead of username and password every time.

Server:
Token ke Header aur Payload ko phir se same Secret Key ke sath combine karke ek naya signature banata hai.

Server creates signature again (using same key)

Agar dono signatures same aaye:
✅ Token valid hai → Request allow hoti hai.
Agar dono signatures match nahi hue:
❌ Token fake / tampered / expired hai → Server 401 Unauthorized de deta hai.

JWT verification me server token ke header aur payload ko same secret key se dubara sign karta hai aur is signature ko 
client ke token ke signature se compare karta hai. Agar dono match ho jaye to token valid, nahi to 401 Unauthorized.

6. JWT Types
Technically JWT ek hi type ka hota hai, lekin use ke hisab se do major scenarios hote hain:
--Access Token -
-Short-lived (5–15 minutes)

Long-lived (days/months)

8. JWT Advantages
✅ Fast (stateless)
✅ Easy to use in web & mobile apps
✅ No server-side session storage required
✅ Secure if HTTPS + proper secret key used
------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------

9. JWT Disadvantages
❌ Once issued, can’t be revoked easily until expired
❌ If secret key leak ho jaye → sab token compromised
❌ Large payload means bigger network transfer

Good question 👍

In Web API (ASP.NET Core or ASP.NET),

👉 Role-based authentication (actually the correct term is Role-based Authorization) means:

A user must first authenticate (prove who they are, usually via username/password, token, etc.).

After authentication, the system checks the roles assigned to that user (like Admin, Manager, User, Student, etc.).

Based on the role, the user is authorized (or denied) to access certain APIs or perform certain actions.

🔹 How it Works

Authentication:
User logs in → System validates credentials → Issues a token (JWT or cookie).
The token contains the user’s claims, including their roles.

Example:

{
  "sub": "12345",
  "name": "Bhavesh",
  "role": "Admin"
}


Authorization:
When user calls an API, Web API checks if the user’s role matches the required role.

🔹 Example in ASP.NET Core Web API
[Authorize(Roles = "Admin")]
[HttpGet("GetAllUsers")]
public IActionResult GetAllUsers()
{
    return Ok("Only Admins can see this!");
}


If the logged-in user has role Admin, they can access this API.

If the role is User, they will get 403 Forbidden.

🔹 Multiple Roles Example
[Authorize(Roles = "Admin,Manager")]
[HttpGet("Reports")]
public IActionResult GetReports()
{
    return Ok("Admins and Managers can view reports");
}


✅ In short:

Authentication = “Who are you?”

Authorization (Role-based) = “What are you allowed to do based on your role?”
